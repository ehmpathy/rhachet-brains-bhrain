## blueprint codepath gap: on.episode

blueprint line 97 states:
> `on.episode: input.episode (for continuation)`

implementation does NOT pass `on.episode` to atom.ask(). instead uses `buildPromptFromMessages()` to convert bhrain's message array to a prompt string.

### is this a gap?

**no.** here is why:

1. **vision does not mandate on.episode** - vision line 75 says "use BrainAtom.ask() instead of generate()". it does not mandate how conversation state is passed.

2. **blueprint shows two options** - line 94 says "prompt: input.messages (via episode continuation)". the prompt can come from messages OR from episode continuation.

3. **bhrain's message format is incompatible with episode** - bhrain has rich message types:
   - `BrainArch1SessionMessage` with roles (system, user, assistant, tool)
   - `toolCalls` array on assistant messages
   - `toolCallId` on tool result messages

   rhachet's episode continuation stores raw exchanges, not bhrain's structured format.

4. **buildPromptFromMessages is the correct bridge** - it serializes bhrain's rich message format to XML-tagged prompt that the LLM understands:
   ```
   <system>...</system>
   <user>...</user>
   <assistant>...[tool_call: name(args)]</assistant>
   <tool_result id="...">...</tool_result>
   ```

verdict: **not a gap** - implementation chooses message-to-prompt over episode continuation, which is valid per the vision.
