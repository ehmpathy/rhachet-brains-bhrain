# self-review: has-pruned-backcompat

## the question

for each backwards-compat concern in the blueprint, ask:
- did the wisher explicitly say to maintain this compatibility?
- is there evidence this backwards compat is needed?
- or did we assume it "to be safe"?

## what the wisher said

from `0.wish.md`:
- update BrainArch1 to operate against rhachet's BrainAtom directly
- no more custom atom plugins

**notably absent:** any mention of backwards compatibility. the wish is to CHANGE, not to preserve.

## backwards-compat concerns in blueprint

### 1. genBrainRepl signature change

**current signature:**
```ts
genBrainRepl({ slug, atom, context })
```

**after change:**
```ts
// atom comes from rhachet discovery, not passed by caller
genBrainRepl({ slug, context })  // or via genContextBrain
```

**is this a backcompat break?** YES. callers currently pass `atom`.

**was backcompat requested?** NO. wish says "update to use BrainAtom directly".

**verdict:** INTENTIONAL INCOMPATIBLE CHANGE. no backcompat shim needed.

### 2. BrainArch1Atom deletion

**already deleted.** this is an incompatible change. callers can't import it.

**was backcompat requested?** NO. wish says "no more custom atom plugins".

**verdict:** INTENTIONAL INCOMPATIBLE CHANGE.

### 3. retained files (invokeBrainArch1, iterateBrainArch1Loop)

**blueprint shows `[○] retain`.**

**why retained?** not for backcompat. retained because their internal behavior doesn't change - they call generateBrainArch1LlmResponse which changes internally.

**is this backcompat?** NO. this is "no change needed", not "preserved for compatibility".

**verdict:** NOT A BACKCOMPAT CONCERN.

## findings

### no backcompat shims found

the blueprint doesn't include:
- re-exports of old types
- adapters for old interface
- deprecated function wrappers
- fallback logic

### incompatible changes are intentional

| change | intentional? |
|--------|-------------|
| BrainArch1Atom removed | YES - wish said "no custom atom plugins" |
| genBrainRepl signature | YES - atom comes from rhachet now |
| atom.generate() → atom.ask() | YES - conform to rhachet |

### no backcompat added "to be safe"

reviewed blueprint for patterns like:
- `// deprecated, kept for compatibility`
- `// TODO: remove in v2`
- fallback to old behavior
- re-exports of removed types

none found.

searched blueprint for backcompat indicators:

2. re-export patterns - none found
3. adapter/wrapper patterns - none found
4. deprecated annotations - none found
5. fallback logic - none found

searched wish/vision for backcompat requests:
1. "maintain compatibility" - not found
2. "backwards compatible" - not found
3. "preserve" - not found
4. "deprecated" - not found

## verification

read blueprint lines 45-102 (filediffs and codepaths) for each file:
- BrainArch1Repl.ts: `[~]` update - intentional change
- generateBrainArch1LlmResponse.ts: `[~]` update - intentional change

## conclusion

no backwards compatibility concerns in blueprint.
