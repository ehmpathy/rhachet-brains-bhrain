### section 4: extension required (lines 25-43)

**blueprint proposes:**
```ts
tools?: ToolDefinition[];        // NEW
toolInvocations?: ToolInvocation[];  // NEW
```

**vision adherance:**
- vision line 84: `tools?: ToolDefinition[]` in input ✓
- vision line 85: `tools?: ToolInvocation[]` in output - REFINEMENT

**deviation analysis:**

vision says:
```
- `tools?: ToolDefinition[]` in input
- `tools?: ToolInvocation[]` in output
```

blueprint says:
```
tools?: ToolDefinition[];        // input
toolInvocations?: ToolInvocation[];  // output
```

the blueprint renames output `tools` to `toolInvocations`.

**is this a valid refinement?**

1. vision line 87: "the exact contract will be specified in the blueprint" - explicitly grants blueprint authority to refine
2. `tools` for both input and output creates confusion
3. `toolInvocations` clearly distinguishes output (invocations made) from input (definitions provided)
4. matches anthropic API pattern where requests have `tools` and responses have `tool_use` blocks

**verdict:** ✓ adheres - refinement is within blueprint's authority per vision line 87
