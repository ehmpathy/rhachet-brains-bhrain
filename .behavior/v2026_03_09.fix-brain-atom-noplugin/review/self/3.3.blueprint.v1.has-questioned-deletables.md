# self-review: has-questioned-deletables

## the question

for each component in the blueprint, ask:
- can this be removed entirely?
- if we deleted this and had to add it back, would we?
- did we optimize a component that shouldn't exist?
- what is the simplest version that works?

## components examined

### 1. message serialization strategy (lines 134-158 in original)

**what it was:** a workaround to serialize messages + tools to a prompt string since BrainAtom.ask() takes prompt, not messages array.

**can this be removed?** YES.

**why:** if bhrain conforms to rhachet, and rhachet doesn't have tool support yet, bhrain should wait. not invent a workaround.

**if we deleted this and had to add it back, would we?** NO. we would ask rhachet to extend BrainAtom instead.

**action taken:** DELETED from blueprint.

### 2. tool use in schema output (lines 160-174 in original)

**what it was:** use zod schema with discriminated union to have LLM emit tool_call vs final_answer.

**can this be removed?** YES.

**why:** this is another workaround. native tool_use API is more efficient and what we actually want.

**if we deleted this and had to add it back, would we?** NO. workaround adds complexity. wait for real solution.

**action taken:** DELETED from blueprint.

### 3. phased approach (lines 202-215 in original)

**what it was:** phase 1 = workaround mode, phase 2 = native mode after rhachet extension.

**can this be removed?** YES.

**why:** two phases only exist because we had workarounds. without workarounds, there's one phase: wait, then implement.

**if we deleted this and had to add it back, would we?** NO. simpler to just wait.

**action taken:** DELETED from blueprint.

### 4. remaining components - do they hold?

#### the gap section
**can be removed?** NO.
**why it holds:** documents what BrainAtom lacks. essential for handoff to rhachet.

#### extension required section
**can be removed?** NO.
**why it holds:** specifies exact contract rhachet needs to implement. this is the handoff doc.

#### filediffs section
**can be removed?** NO.
**why it holds:** declares what changes after rhachet provides extension. necessary for future implementation.

#### codepaths section
**can be removed?** NO.
**why it holds:** shows how code flows change. aids comprehension.

#### contracts section
**can be removed?** NO.
**why it holds:** declares the interfaces bhrain conforms to. essential reference.

#### test updates section
**can be removed?** NO.
**why it holds:** ensures coverage after changes. required by stone template.

#### blocked by section
**can be removed?** NO.
**why it holds:** explicit dependency declaration. core to the blueprint.
