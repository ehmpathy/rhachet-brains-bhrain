## requirement 3: change interface from generate() to ask()

**who said this?** wisher implied it with "use ask"

**CONCERN: this is where I pause.**

the vision correctly identifies a fundamental interface mismatch:

| what BrainArch1 needs | what BrainAtom.ask() provides |
|-----------------------|-------------------------------|
| messages array        | single prompt string          |
| tool definitions      | schema for output             |
| tool call responses   | structured output             |
| raw LLM response      | parsed JSON output            |

**these are different abstraction levels.**

- `ask()` is high-level: prompt in, structured data out
- `generate()` is low-level: messages + tools in, raw response out

an agentic loop needs the low-level interface to:
1. maintain conversation history
2. detect tool calls
3. execute tools
4. feed results back
5. continue the loop

**question for wisher:**
does rhachet's `BrainAtom` have a lower-level method beyond `ask()`?

if not, the options are:
1. add `generate()` or similar to rhachet's BrainAtom
2. bhrain wraps BrainAtom with an adapter (defeats the "no plugins" goal)
3. rethink BrainArch1 to work with `ask()` (major refactor)

---

## what was missed?

the vision assumes BrainAtom can support the agentic loop. this assumption needs validation before we proceed.

## recommendation

before implementation, validate:
- what methods does `BrainAtom` actually expose?
- is there a lower-level method for tool-based conversation?
- if not, what's the path forward?

this is a **blocker** - we cannot proceed until we know the interface.
