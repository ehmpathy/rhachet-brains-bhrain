# self-review: has-idempotent-mutations

## the question

for mutations in the blueprint, ask:
- are state changes idempotent (safe to retry)?
- do mutations use findsert/upsert patterns?
- are side effects guarded against re-entry?

## mutations in blueprint

### 1. BrainAtom.ask() call

**is this a mutation?** NO.

**analysis:**
- ask() is a read operation (inference call)
- no state is mutated in bhrain
- atom state is managed by rhachet

**verdict:** not a mutation. n/a.

### 2. episode continuation

**is this a mutation?** YES. adds exchanges to episode.

**idempotent?** YES.

**analysis:**
- episode is append-only
- same prompt + same episode = same result (LLM determinism aside)
- re-entry produces same episode state

**verdict:** mutation is idempotent.

### 3. tool execution

**is this a mutation?** YES. tools may mutate external state.

**idempotent?** DEPENDS on tool.

**analysis:**
- tool idempotency is tool author's responsibility
- BrainArch1 loop does not add idempotency guards
- marked `[○]` retain - no change in blueprint

**verdict:** tool idempotency unchanged. out of scope.

## findings

### no new mutations

blueprint does not introduce new mutations:
- type changes only
- behavioral change is method call (generate→ask)
- no new state management

### extant mutations are idempotent

| mutation | pattern | idempotent? |
|----------|---------|-------------|
| session append | append-only | yes |
| episode append | append-only | yes |

### no findsert/upsert needed

blueprint has no database or persistence mutations. findsert/upsert patterns not applicable.

## conclusion

mutations in blueprint are idempotent.

no new mutations introduced. extant mutations use append-only patterns which are inherently idempotent. tool execution idempotency is out of scope (tool author responsibility).
