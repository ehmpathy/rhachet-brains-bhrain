## deeper reflection

### what is the `.play.test.ts` convention?

the `.play.test.ts` convention distinguishes journey tests from unit/integration tests:

| suffix | purpose | scope |
|--------|---------|-------|
| `.test.ts` | unit tests | isolated functions |
| `.integration.test.ts` | integration tests | multiple components |
| `.play.test.ts` | journey tests | user-experienced flows |

journey tests are "playbooks" — they replay what a user does.

### why not adopt `.play.test.ts` now?

1. **repo-wide convention change** — would require rename of extant integration tests
2. **out of scope** — this is an internal refactor, not a test convention refactor
3. **no new journeys** — no new files to name correctly

### when would `.play.test.ts` matter?

| scenario | action |
|----------|--------|
| new repo | establish `.play.test.ts` from start |
| new user journey | consider `.play.test.ts` if repo supports |
| internal refactor | use extant convention |

this behavior falls in the last row.

### lessons learned

#### 1. conventions serve purpose, not themselves

the `.play.test.ts` convention exists to distinguish journey tests. if a repo uses `.integration.test.ts` for the same purpose, the convention is served even if the name differs.

#### 2. n/a requires verification, not assumption

i verified by:
1. ran `git ls-files '*.test.ts'` — confirmed no `.play.test.ts` files
2. read extant test patterns — confirmed `.integration.test.ts` serves journey purpose
3. checked scope — confirmed internal refactor, no new journeys

#### 3. scope discipline prevents scope creep

the temptation: "let's adopt `.play.test.ts` convention while we're here."

the discipline: this behavior is about BrainAtom contract conformance. test file conventions are a separate behavior. stay in scope.
