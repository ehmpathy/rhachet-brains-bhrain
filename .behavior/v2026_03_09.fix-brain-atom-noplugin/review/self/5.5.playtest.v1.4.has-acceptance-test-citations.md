## deeper reflection

### why steps 1, 2, 3, 6 lack automated tests

these steps verify **structural properties**, not **runtime behavior**:

| step | verifies | why shell command is appropriate |
|------|----------|----------------------------------|
| 1 | directory deletion | filesystem state, not code behavior |
| 2 | import source | code structure, not runtime |
| 3 | absence of imports | code structure, not runtime |
| 6 | method call name | code structure, not runtime |

automated tests verify runtime behavior. structural verification is done via:
- shell commands (grep, ls) for human playtests
- linters or static analysis for CI (not implemented here)

**gap assessment**: these are NOT gaps. structural verification belongs in manual playtests or CI linters, not runtime tests.

### why integration tests serve as acceptance tests

this repo has no separate acceptance test suite because:
1. bhrain is a library, not a user-visible application
2. the "user" is a developer who imports and uses the library
3. integration tests exercise the full public API (invokeBrainArch1)
4. there is no UI or CLI to acceptance-test

**the integration tests ARE the acceptance tests** for this library.

### playtest vs automation coverage

| coverage type | mechanism |
|---------------|-----------|
| structural (files, imports, methods) | playtest shell commands |
| behavioral (API works correctly) | integration tests |
| full journey (end-to-end) | integration tests (cases 1, 2, 5) |

the playtest combines both: shell commands for structure, test suite for behavior.

## review findings

### non-issues (holds)

#### 1. no dedicated acceptance tests — holds

**why it holds**: bhrain is a library. the "acceptance criteria" is "does the API work correctly?" — integration tests answer this question. a separate acceptance test suite would duplicate coverage.

#### 2. structural steps lack automated tests — holds

**why it holds**: steps 1, 2, 3, 6 verify code structure, not runtime behavior. shell commands are the correct tool. automated structural verification would require a linter or static analysis — out of scope for this behavior.

#### 3. all cited test cases exist — holds

**why it holds**: verified via grep. cases 1, 2, 5 exist at the documented lines. case 4 (openai tool) is `given.skip` as documented.

### found issues

none found. the playtest correctly cites automated tests where they exist and uses shell commands where automation is inappropriate.

### lessons learned

#### 1. acceptance tests are for applications, not libraries

for libraries, integration tests ARE acceptance tests. the "user" is a developer, and the "acceptance criteria" is "does the API work?"

#### 2. structural verification != behavioral verification

a directory delete or import change is structural. a call to `atom.ask()` that retrieves a response is behavioral. different verification tools for different concerns.

#### 3. playtest = manual verification + automated tests

the playtest is not redundant with tests. it combines:
- structural checks humans can verify (shell commands)
- behavioral checks tests verify (cited test cases)
