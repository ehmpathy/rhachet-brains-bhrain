## deeper reflection

### when are repros required?

repros capture user experience before code exists. they're needed when:
- new user journeys are introduced
- extant journeys change significantly
- edge cases need documentation

repros are NOT needed when:
- internal implementation changes without user-faced impact
- contracts remain identical
- users cannot observe any difference

### why this refactor didn't need repros

the user experience is unchanged:

**before refactor:**
```ts
const brain = await genBrainRepl({ slug: 'bhrain/arch1', atom, ... });
const response = await brain.ask({ prompt: '...' });
```

**after refactor:**
```ts
const brain = await genBrainRepl({ slug: 'bhrain/arch1', atom, ... });
const response = await brain.ask({ prompt: '...' });
```

identical. the internal plumbing changed (`BrainArch1Atom` → `BrainAtom`), but the public surface stayed the same.

### the lesson: repros follow journeys, not code

repros document user journeys. journeys are user-observed sequences of actions and outcomes.

internal refactors change code paths, not journeys. therefore:
- no new journeys = no new repros
- preserved journeys = preserved extant tests

the extant integration tests (`invokeBrainArch1.integration.test.ts`) are the journey tests. they existed before this refactor and continue to pass after. that's the evidence that journeys were preserved.

### what if i had incorrectly created repros?

to create repros for an internal refactor would be inaccurate:
- implies new user-faced capability
- creates false documentation
- suggests the user should expect behavioral changes

accurate documentation requires us to acknowledge when there's no user-faced change.
