## deeper reflection

### why do snapshots matter?

snapshots serve two purposes:
1. **vibe check in PRs**: reviewers see actual output without execution
2. **drift detection**: output changes surface in diffs

for public contracts, snapshots capture what the caller sees. this is valuable because:
- reviewers can verify output looks correct
- accidental changes show up as snapshot diffs
- edge cases are documented by example

### why no snapshots for internal refactors?

internal refactors change code paths, not outputs. if the public contract is unchanged, then:
- callers see the same response shapes
- error messages remain identical
- edge case behavior is preserved

new snapshots would be redundant. the extant snapshots (if any) already cover the public surface.

### what would have required snapshots?

| change type | requires snapshots? | example |
|-------------|--------------------|---------|
| new CLI command | yes | new stdout format to verify |
| new SDK method | yes | new response shape to verify |
| changed error message | yes | error format changed |
| internal refactor | no | output unchanged |

this behavior is the last row: internal refactor, output unchanged.

### lessons learned

#### 1. snapshots protect boundaries, not internals

the purpose of snapshots is to verify what crosses a boundary:
- CLI → user (stdout/stderr)
- SDK → caller (response shape)
- API → client (HTTP response)

internal boundaries don't need snapshots. the tests themselves verify internal behavior.

#### 2. absence of snapshots can be correct

"no snapshots needed" is a valid conclusion when:
- no new public contracts exist
- extant contracts are unchanged
- changes are purely internal

to force snapshots for internal changes would create noise:
- snapshots would drift on every refactor
- reviewers would ignore them
- value of snapshots would degrade

#### 3. the question to ask: "what does the caller see?"

for any change, ask: "does this change what the caller observes?"

| answer | action |
|--------|--------|
| yes, new shape | add snapshot |
| yes, changed shape | update snapshot |
| no, same shape | no snapshot needed |

this refactor: the caller sees the same genBrainRepl, same ask(), same response shape. answer is "no" → no snapshot needed.

#### 4. verify the claim "unchanged"

"public contract unchanged" is a claim that must be verified. i verified by:
1. read the public API surface before and after
2. confirm input and output types are identical
3. run extant tests to confirm behavior unchanged

without verification, "unchanged" would be an assumption, not a fact.
