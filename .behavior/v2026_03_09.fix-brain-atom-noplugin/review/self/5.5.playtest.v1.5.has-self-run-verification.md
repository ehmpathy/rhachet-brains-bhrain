### lessons learned

#### 1. self-test proves playtest accuracy

by executing each step, I verified that:
- commands are syntactically correct
- paths exist (or correctly don't exist)
- expected outputs match actual outputs
- tests actually pass (not just exist)

this is more valuable than review alone. a review can verify that instructions "look right" but only execution proves they work.

**concrete example**: step 2 expects imports at lines 4 and 5. only execution verified that:
- the file exists at the specified path
- the grep command finds the pattern
- the line numbers are exactly 4 and 5 (not 3 and 4, or 5 and 6)

#### 2. API keys via skill enable full execution

the `use.apikeys.sh` skill loads credentials from `~/.config/rhachet/apikeys.env`. this enabled full execution of integration tests without manual env setup.

**key command**: `source .agent/repo=.this/role=any/skills/use.apikeys.sh && npm run test:integration`

this removed the only blocker to complete self-execution. no steps deferred.

#### 3. documentation vs execution

the playtest is a documentation artifact AND an execution guide. self-test validates both:
- documentation: instructions are clear and complete
- execution: commands produce expected results

**the key insight**: a playtest that cannot be self-executed is a playtest that may fail for the foreman. self-execution is the quality gate.

#### 4. exact vs approximate expectations

the playtest uses exact expectations where possible:
- exact error message for step 1
- exact line numbers for step 2
- exact method names for step 6
- exact test case names for step 5

this makes pass/fail unambiguous. when I ran the commands, the outputs matched exactly — no interpretation needed.
