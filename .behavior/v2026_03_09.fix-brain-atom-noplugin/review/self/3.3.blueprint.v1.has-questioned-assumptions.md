# self-review: has-questioned-assumptions

## the question

for each technical assumption in the blueprint, ask:
- what do we assume here without evidence?
- what if the opposite were true?
- is this architecture choice based on evidence or habit?
- what exceptions or counterexamples exist?
- could a simpler approach work?

## assumptions examined

### 1. BrainAtom WILL be extended with tools

**what we assume:** rhachet will extend BrainAtom.ask() with `tools` input and `toolInvocations` output.

**evidence?** none. this is a handoff request, not a confirmed plan.

**what if the opposite were true?** if rhachet decides NOT to extend BrainAtom, bhrain remains blocked forever. or needs to find alternative.

**verdict:** assumption is RISKY but ACCEPTABLE. rhachet owns the contract. we can only request. if rejected, we pivot. but this is the correct ask given "bhrain conforms to rhachet".

### 2. episode continuation can handle tool flow

**what we assume:** `on.episode` with BrainExchange[] can represent tool execution history.

**evidence?** BrainExchange has `input: string` and `output: string`. tool results would need to fit this model.

**what if the opposite were true?** if episodes can't represent tool flow, a new primitive is needed.

**counterexample:** current BrainExchange is simple input/output pairs. tool execution has more structure:
- tool call (name, args)
- tool result (output, error?)
- multi-tool in single turn

**verdict:** assumption is QUESTIONABLE. BrainExchange may need extension too, not just BrainAtom. ADD TO HANDOFF.

### 3. type change is "simple"

**what we assume:** the filediffs say `[~]` (update) for type change from BrainArch1Atom → BrainAtom.

**evidence?** looked at both interfaces:
- BrainArch1Atom.generate({ messages, tools }) → { message, tokenUsage }
- BrainAtom.ask({ prompt, schema }) → BrainOutput

these are NOT compatible. generate() takes messages array. ask() takes prompt string.

**what if the opposite were true?** the "simple type change" is actually a fundamental refactor of how BrainArch1 calls its atom.

**verdict:** assumption is WRONG. this is not a type change. it's a behavior change. UPDATE BLUEPRINT.

### 4. genContextBrain handles composite slug

**what we assume:** rhachet's `genContextBrain({ choice: { repl: 'bhrain/arch1@claude/sonnet' } })` will discover bhrain's repl and compose it with the atom.

**evidence?** verified in prior session via code inspection of rhachet's genContextBrain.d.ts.

**verdict:** assumption HOLDS based on evidence.

### 5. blocked = wait (no alternative)

**what we assume:** since we're blocked by rhachet, we just wait.

**what if the opposite were true?** we could:
- contribute the extension to rhachet ourselves
- propose an RFC to rhachet
- identify interim solution that doesn't require workaround

**verdict:** assumption is LAZY but ACCEPTABLE. a contribution to rhachet is out of scope for this behavior. the handoff is the correct next step.
