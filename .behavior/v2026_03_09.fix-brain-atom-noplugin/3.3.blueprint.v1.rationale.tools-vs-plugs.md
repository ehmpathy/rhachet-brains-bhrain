# rationale: plugs.tools and calls.tools

> design decision

## the pattern

```ts
ask({
  prompt,
  schema,
  plugs: {
    tools: ToolDefinition[]  // INPUT: plug tools into the brain
  }
}) => Promise<BrainOutput & {
  calls: {
    tools: ToolInvocation[]  // OUTPUT: tools the LLM chose to invoke
  }
}>
```

## why plugs.tools

**tools are plugins. you plug tools into the brain to use them.**

plug is the right place:
- semantic fit: tools are "plugged in"
- extensible: plug can hold other pluggable resources later
- clean: keeps top-level params focused on the prompt/schema

## why calls.tools

tool calls are OUTPUT - the result of the LLM's decision to use tools.

```ts
output.calls.tools: ToolInvocation[]
```

nested under `output.call` for:
- symmetry: `plugs.tools` in, `calls.tools` out
- namespace: keeps call-related output grouped
- extensibility: `output.call` can hold other call metadata later
- clarity: `output.calls.tools` reads naturally

## the contract

### input (in plug)

```ts
plugs: {
  tools?: ToolDefinition[]
}
```

### output (on output)

```ts
output: {
  // ... extant fields
  calls?: {
    tools: ToolInvocation[]
  }
}
```

## summary

| concern | location | why |
|---------|----------|-----|
| tool definitions | `plugs.tools` | tools are plugins |
| tool invocations | `calls.tools` | calls are output |
