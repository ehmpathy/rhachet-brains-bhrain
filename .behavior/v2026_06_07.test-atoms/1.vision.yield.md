# vision: test m2.5 and m2.7 on codewrite

## the outcome world

### day-in-the-life

**before:**
- developer runs codewrite tests with xai/grok-code-fast-1
- test fails: "model deprecated"
- scramble to find replacement cheapfast code brain
- no clear benchmark data on alternatives

**after:**
- developer runs `npm test -- --grep "fireworksai.codewrite"`
- tests use m2.5 and m2.7 via `rhachet-brains-fireworksai`
- pass/fail results show which model works for code generation
- clear evidence for cheapfast brain atom selection

### aha moment

when the test generates a working shell command using m2.5 at $0.3/1M input — ~10x cheaper than grok-code-fast-1 was — and it works on the first try.

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| find grok-code-fast-1 replacement | partial | tests will show pass/fail, but need to run them |
| evaluate m2.5 for code | yes | codewrite test exercises code + tool use |
| evaluate m2.7 for code | yes | same test, different slug |
| compare cost/performance | partial | cost is known ($0.3/1M), perf needs test run |

### pros

- **low cost**: both models at $0.3/1M input, $1.2/1M output
- **fast**: m2.5 has 0.76s TTFT, m2.7 at 70 tok/s
- **tool use**: both support function calling natively
- **extant pattern**: test structure already extant from xai/chutes tests

### cons

- **m2.5 reasoning limits**: 42/100 on intelligence index — may struggle with complex logic
- **context handling**: m2.7 slower TTFT (1.08s) vs m2.5 (0.76s)
- **untested territory**: no prior data on these models with our briefs

### edge cases + pit of success

| edge case | handling |
|-----------|----------|
| model hallucinates API endpoint | network sandbox blocks unauthorized domains |
| model doesn't self-test | test harness verifies output anyway |
| model exceeds token limit | maxTokens=128000 matches model context |
| API key absent | keyrack.source() fails fast with clear error message |

## what is awkward?

### naming inconsistency

- wish says "m2.5 and m2.7"
- package slugs are `fireworks/minimax/2.5` and `fireworks/minimax/2.7`
- official model names are "MiniMax-M2.5" and "MiniMax-M2.7"
- slightly confusing but follows extant slug pattern

### test file proliferation

- already have: xai.codewrite, chutes.codewrite
- adding: fireworksai.codewrite
- all nearly identical except for atom/slug
- could refactor to shared test with parameterized atoms, but that's scope creep

### cheapfast vs best

- m2.5 is "cheapfast" but may struggle with complex briefs
- m2.7 is stronger but slower
- wish wants "best cheapfast" — these may be in tension
- test will reveal which (if either) works for our use case

### model recommendation nuance

per research, m2.5 works best when:
> "Use stronger reasoning models (Kimi K2.5, DeepSeek) for logic/architecture, then hand spec to M2.5 for code generation"

our codewrite test gives it a spec directly — this should be m2.5's sweet spot. but it also gets mechanic briefs (~50K tokens) which may tax its reasoning limits.
