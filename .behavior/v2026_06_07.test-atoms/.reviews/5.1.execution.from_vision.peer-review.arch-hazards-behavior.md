# blocker.1: Order-dependent credential check before keyrack sourcing

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.behavior-hazards.md

**locations**:
- jest.acceptance.env.ts:15
- jest.integration.env.ts:25

The AWS credential verification runs before the keyrack.source call that is intended to inject credentials from .agent/keyrack.yml into process.env. Since top-level module code executes in source order, if declapract.use.yml indicates awsAccountId but no AWS_* env vars are present yet, it throws 'no aws credentials present' before keyrack can source them. This causes tests to fail to start even when credentials are available via keyrack. The behavior depends on the order of top-level statements and external state at module load time, leading to unpredictable test execution. Matches order-dependence hazard.

**snippet**:
```ts
const declapractUsePath = join(process.cwd(), 'declapract.use.yml');
const requiresAwsAuth =
  existsSync(declapractUsePath) &&
  readFileSync(declapractUsePath, 'utf8').includes('awsAccountId');
if (
  requiresAwsAuth &&
  !(process.env.AWS_PROFILE || process.env.AWS_ACCESS_KEY_ID)
)
  throw new Error(
    'no aws credentials present. please authenticate with aws to run acceptance tests',
  );

/**
 * .what = source credentials from keyrack for test env
 * .why = auto-inject keys into process.env, fail fast if absent
 */
import { keyrack } from 'rhachet/keyrack';

const keyrackYmlPath = join(process.cwd(), '.agent/keyrack.yml');
if (existsSync(keyrackYmlPath)) {
  keyrack.source({
    env: 'test',
    owner: 'ehmpath',
    mode: 'strict',
  });
}
```

This is a classic non-determinism behavior hazard.
