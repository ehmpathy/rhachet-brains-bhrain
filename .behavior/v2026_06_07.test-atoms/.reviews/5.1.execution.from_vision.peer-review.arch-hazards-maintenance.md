# blocker.1: failhide: try/catch swallows execSync errors in tool executions

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.maintenance-hazards.md

**locations**:
- src/domain.operations/arch1/core/invokeBrainArch1.fireworksai.deepseek-v4-flash.codewrite.integration.test.ts
- src/domain.operations/arch1/core/invokeBrainArch1.fireworksai.m25.codewrite.integration.test.ts
- src/domain.operations/arch1/core/invokeBrainArch1.fireworksai.m27.codewrite.integration.test.ts
- src/domain.operations/arch1/core/invokeBrainArch1.xai.codewrite.integration.test.ts

The execute functions for 'test' and 'chmod' tools wrap execSync and fs.chmod in try/catch, catching all errors (including unexpected ones) and returning structured failure info instead of rethrowing. Per the rule, catch must only allowlist expected errors; unexpected errors must rethrow. Silent swallowing (or redirecting to return values) hides real errors, causes silent defects, and makes debugging hard. This pattern appears identically across code generation tests.

**snippet**:
```ts
              try {
                const stdout = execSync(invocation.input.command, {
                  encoding: 'utf-8',
                  cwd: outputDir,
                  timeout: 30000,
                  env: sandboxedEnv,
                });
                return { stdout, stderr: '', exitCode: 0 };
              } catch (error: unknown) {
                const execError = error as {
                  stdout?: string;
                  stderr?: string;
                  status?: number;
                };
                return {
                  stdout: execError.stdout ?? '',
                  stderr: execError.stderr ?? String(error),
                  exitCode: execError.status ?? 1,
                };
              }
```

# blocker.2: failhide: bare catch hides database connection errors

**rule**: .agent/repo=bhuild/role=behaver/briefs/practices/behavior.execution/architect/rule.forbid.maintenance-hazards.md

**locations**:
- jest.integration.env.ts

The psql execSync check uses a bare catch {} that discards the original error and throws a generic message. This violates failhide (unexpected errors must rethrow) and failloud (errors must include actionable context like the actual failure reason, config values, and cause). Real connection errors are hidden, leading to debug hours.

**snippet**:
```ts
  try {
    execSync(
      `PGPASSWORD="${testConfig.database.role.crud.password}" psql -h ${testConfig.database.tunnel.local.host} -p ${testConfig.database.tunnel.local.port} -U ${testConfig.database.role.crud.username} -d ${testConfig.database.target.database} -c "SELECT 1" > /dev/null 2>&1`,
      { timeout: 3000 },
    );
  } catch {
    throw new Error(
      `did you forget to \`npm run start:testdb\`? cant connect to database`,
    );
  }
```

# blocker.4: immutability: use of let instead of const

**locations**:
- src/domain.operations/arch1/core/invokeBrainArch1.togetherai.integration.test.ts

let testDir is declared and reassigned in beforeAll/afterAll. The rule forbids let/var except in explicitly scoped mutation blocks; all bindings must be const with immutable updates via spread/clone. Mutable state here risks test pollution and concurrency issues.
