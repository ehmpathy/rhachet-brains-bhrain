# vision: eject BrainArch1 into rhachet-brains-bhrain

> the world where bhrain has its own brain supplier package

---

## the outcome world

### day-in-the-life with this in place

a developer who wants agentic workflows with rhachet installs one package:

```bash
npm install rhachet-brains-bhrain
```

they get a BrainRepl that:
- conforms to rhachet's standard BrainRepl interface
- orchestrates multi-turn agentic loops with tool use via BrainArch1 internally
- uses composite slug `bhrain/arch1@$atom` to specify which atom powers it
- supports ask (readonly) and act (read+write) modes

**composite slug pattern** — `$repl@$atom` bakes the atom choice into the repl:
```ts
import { genContextBrain } from 'rhachet/brains';

// discover repl+atom via composite slug
const brain = await genContextBrain({ choice: 'bhrain/arch1@claude/sonnet' });

// use directly — atom is baked in
const result = await brain.ask({ prompt, schema, role });
```

they use it like:

```ts
import { genContextBrain } from 'rhachet/brains';
import { z } from 'zod';

// create a brain.repl instance with atom baked in
const brain = await genContextBrain({ choice: 'bhrain/arch1@claude/sonnet' });

// readonly analysis (ask mode)
const analysis = await brain.ask({
  role: { briefs: [reviewerBrief] },
  prompt: 'analyze this codebase and suggest improvements',
  schema: { output: z.object({ suggestions: z.array(z.string()) }) },
});
console.log(analysis.output.suggestions);

// read+write actions (act mode)
const result = await brain.act({
  role: { briefs: [mechanicBrief] },
  prompt: 'fix the bug in src/utils.ts',
  schema: { output: z.object({ fixed: z.boolean(), changes: z.string() }) },
});
console.log(result.output.changes);
```

### before/after contrast

| before | after |
|--------|-------|
| BrainArch1 lives in rhachet-roles-bhrain | BrainArch1 lives in rhachet-brains-bhrain |
| role repo has brain implementation mixed with role behaviors | role repo focuses on roles; brain repo focuses on brains |
| confusing ownership — is it a role or a brain? | clear ownership — brains in brains, roles in roles |
| harder to reuse BrainArch1 in other contexts | any project can npm install the brain |

### the "aha" moment

the value clicks when:
- a developer realizes they can use bhrain's agentic brain without pulling in all the role machinery
- the separation makes it obvious that brains are reusable infrastructure, not role-specific behavior
- tests run faster because the brain can be tested in isolation

## user experience

### usecases

| usecase | goal |
|---------|------|
| **agentic coding** | developer wants an agent that can read, think, execute tools, and iterate |
| **sandboxed exploration** | developer wants to explore a codebase safely without writes |
| **multi-turn workflows** | developer needs conversation continuity across multiple prompts |
| **role-agnostic brains** | developer wants the brain without bhrain's specific role configuration |

### contract inputs & outputs

**public interface (BrainRepl from rhachet):**

inputs to `brain.ask()` / `brain.act()`:
- `on`: (optional) episode or series for continuation
- `plugs`: (optional) repl-specific plugins
- `role.briefs`: briefs to include in system prompt
- `prompt`: the user prompt
- `schema.output`: zod schema for structured output

outputs (`BrainOutput<TOutput, 'repl'>`):
- `output`: structured output that conforms to schema
- `metrics.size.tokens`: input/output/cache token counts
- `metrics.cost.cash`: dollar cost
- `metrics.cost.time`: elapsed milliseconds
- `episode`: continuation handle for multi-turn
- `series`: continuation handle for multi-session

**internal implementation (BrainArch1):**

BrainArch1 is the internal agentic loop. not exposed via SDK, but powers the BrainRepl.

- `BrainArch1Repl`: configures atom, toolboxes, constraints, memory, permission
- `invokeBrainArch1`: entry point for the loop
- `BrainArch1LoopResult`: internal result with iterations, messages, token usage

**key architecture decision: composite slug `$repl@$atom` bakes atom into repl**

```
┌─────────────────────────────────────────────────────────┐
│ rhachet-brains-bhrain                                   │
│                                                         │
│   genBrainRepl({ slug: 'bhrain/arch1@claude/sonnet' }) │
│     ├── parses: repl=bhrain/arch1, atom=claude/sonnet  │
│     └── BrainArch1Repl                                 │
│           └── atom: BrainAtom  ←─────────────────────┐ │
│                                                       │ │
└───────────────────────────────────────────────────────│─┘
                                                        │
┌───────────────────────────────────────────────────────│─┐
│ rhachet-brains-anthropic                              │ │
│   genBrainAtom({ slug: 'claude/sonnet' }) ────────────┘ │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ rhachet-brains-openai                                   │
│   genBrainAtom({ slug: 'gpt-4o' })                      │
└─────────────────────────────────────────────────────────┘
```

BrainArch1Repl accepts any `BrainAtom` — it does NOT wrap SDKs directly. atoms come from:
- `rhachet-brains-anthropic` for claude models
- `rhachet-brains-openai` for openai models
- other brain supplier packages as they become available

### what it looks like

```ts
import { genContextBrain } from 'rhachet/brains';
import { z } from 'zod';

// composite slug: bhrain/arch1@claude/sonnet
const brain = await genContextBrain({ choice: 'bhrain/arch1@claude/sonnet' });

// simple ask (readonly)
const analysis = await brain.ask({
  role: { briefs: [] },
  prompt: 'what does this function do?',
  schema: { output: z.object({ explanation: z.string() }) },
});
console.log(analysis.output.explanation);

// continuation via episode
const step1 = await brain.ask({
  role: { briefs: [] },
  prompt: 'read the config file',
  schema: { output: z.object({ content: z.string() }) },
});
const step2 = await brain.ask({
  on: { episode: step1.episode },  // pass episode for continuation
  role: { briefs: [] },
  prompt: 'now suggest improvements',
  schema: { output: z.object({ suggestions: z.array(z.string()) }) },
});

// act mode for write operations
const fix = await brain.act({
  role: { briefs: [mechanicBrief] },
  prompt: 'fix the type error in utils.ts',
  schema: { output: z.object({ fixed: z.boolean(), diff: z.string() }) },
});
console.log(fix.output.diff);
```

### timeline

1. **install**: `npm install rhachet-brains-bhrain`
2. **configure**: import and instantiate the brain
3. **ask**: send prompts, get responses
4. **iterate**: continue conversations, refine outputs

## mental model

### how users describe it to a friend

> "it's like claude code but as a library — you get the agentic brain that can read files, run commands, and think through problems, but you control when and how it's invoked"

### analogies

| concept | analogy |
|---------|---------|
| brain.atom | a calculator — you input, it outputs, no memory |
| brain.repl | a colleague — you have a conversation, they remember context, they can take action |
| BrainArch1 | a colleague with a specific methodology — bhrain's approach to agentic coordination |

### terms they use vs we use

| user term | public term | internal term | definition |
|-----------|-------------|---------------|------------|
| "agent" | BrainRepl | BrainArch1Repl | configured repl that can think and act |
| "model" | (via slug) | BrainAtom | from rhachet-brains-* packages |
| "context" | episode | conversationHistory | prior state for continuation |
| "readonly" | brain.ask() | permissionGuard | readonly analysis mode |
| "write" | brain.act() | permissionGuard | read+write action mode |

## evaluation

### how well does it solve the goals?

| goal | solved? | how |
|------|---------|-----|
| separation of concerns | ✅ | brains in brains repo, roles in roles repo |
| reusability | ✅ | any project can import the brain package |
| testability | ✅ | brain can be tested in isolation |
| clear ownership | ✅ | one package owns one capability |

### pros

- **modularity**: brain is independent, reusable infrastructure
- **clarity**: obvious where brain logic lives
- **versioning**: brain can evolve independently of roles
- **testing**: faster, focused test suites

### cons

- **dependency management**: must coordinate versions between brain and role repos
- **migration effort**: code currently importing from roles repo needs updating

### edgecases and pit of success

| edgecase | pit of success strategy |
|----------|------------------------|
| no permission guard | default allowAll guard permits all operations |
| input exceeds token limit | fail-fast with BadRequestError before loop starts |
| max iterations reached | terminationReason = MAX_ITERATIONS signals forced stop |
| empty user input | returns helpful message to request input |

## open questions & assumptions

### assumptions

1. **BrainArch1 is stable** — the implementation is mature enough to extract
2. **minimal coupling** — BrainArch1 doesn't deeply depend on role-specific code
3. **100 files accurate** — the file inventory in the handoff is complete
4. **worktree extant** — the FROM directory (`rhachet-roles-bhrain.vlad.eject-brain-arch1`) is present and ready

### questions to validate

1. **dependencies**: does BrainArch1 require any packages not yet in this repo's deps?
2. **imports**: are there circular dependencies that will break when split?
3. **tests**: do all tests pass standalone, or do some require role-repo fixtures?
4. **naming**: is `rhachet-brains-bhrain` the final package name? (currently the repo name matches but package.json says anthropic)

---

## what is awkward?

### what feels off

- **rename anthropic → bhrain**: the current repo has anthropic-specific code that we're removing and replacing with bhrain-specific code — feels like a transformation more than an ejection
- **100 files moved**: large file move increases risk of broken imports or missed references

### where design fights mental model

- **"brains-bhrain"**: the name is a bit recursive — "the bhrain brains package" — but it follows the convention (`rhachet-brains-{supplier}`)

### uncomfortable tradeoffs

| tradeoff | discomfort | justification |
|----------|------------|---------------|
| depends on rhachet-brains-* | adds external dependencies | enables code reuse; atoms already tested; avoids duplication |
| large move | 96 files (68 src + 28 briefs) | one-time migration cost; cleaner separation pays off long-term |
| coordination needed | roles repo must update imports | necessary for proper separation |

## summary

after this ejection:

```
rhachet-brains-bhrain/
├── src/
│   ├── domain.objects/BrainArch1/         # [internal] brain architecture types
│   ├── domain.operations/arch1/            # [internal] brain loop implementation
│   ├── domain.operations/brain.repl/       # [+] BrainRepl wrapper (uses BrainAtom directly)
│   └── contract/sdk/                       # public exports (genBrainRepl only)
├── src/index.ts                            # exposes only BrainRepl interface
└── .agent/repo=.this/role=architect/briefs/ # architect briefs (28 files)
```

**no direct SDK dependencies** — atoms come from:
- `rhachet-brains-anthropic` (dependency)
- `rhachet-brains-openai` (dependency)

**public api** (what consumers use):
- `genBrainRepl({ slug })` — creates a BrainRepl
- `brain.ask()` — readonly analysis
- `brain.act()` — read+write actions

**internal** (not exposed):
- BrainArch1Repl, invokeBrainArch1, etc.

developers get:
- `npm install rhachet-brains-bhrain`
- a BrainRepl that conforms to rhachet's standard interface
- powered by BrainArch1 internally, but abstracted away
