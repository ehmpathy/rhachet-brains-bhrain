## divergences found

### divergence 1: invoke* prefix

**codebase pattern:** no `invoke*` functions
**blueprint:** `invokeBrainArch1`

**question:** should this be `genBrainArch1` to match codebase `gen*` pattern?

**analysis:**
- `gen*` = generator functions that create and return objects
- `invoke*` = entry point functions that execute operations
- `invokeBrainArch1` DOES something (runs brain loop), doesn't just return a config

**verdict:** ACCEPTABLE — `invoke*` is semantically correct for an entry point operation

### divergence 2: toolbox* prefix

**codebase pattern:** no `toolbox*` exports
**blueprint:** `toolboxFiles`, `toolboxBash`, `toolboxWeb`

**question:** should these follow a codebase pattern?

**analysis:**
- codebase has no equivalent concept
- toolboxes are BrainArch1-specific tool collections
- naming is self-descriptive

**verdict:** ACCEPTABLE — new concept warrants new prefix

### divergence 3: directory structure

**codebase pattern:** `domain.operations/{atoms,repls,hooks}/`
**blueprint:** `domain.operations/arch1/{core,loop,plugins,...}/`

**question:** should arch1 flatten to `domain.operations/arch1-core/`, `domain.operations/arch1-plugins/`?

**analysis:**
- arch1 is a self-contained module with multiple concerns
- nested structure keeps related code together
- flat structure would scatter arch1 across many top-level directories

**verdict:** ACCEPTABLE — nested structure is appropriate for module cohesion
