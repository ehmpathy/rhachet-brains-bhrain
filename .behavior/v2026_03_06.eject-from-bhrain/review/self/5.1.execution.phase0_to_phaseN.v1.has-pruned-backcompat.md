# self-review: has-pruned-backcompat

## context

this is an ejection — a NEW package (rhachet-brains-bhrain) was created by code extracted from rhachet-roles-bhrain. there is no prior version of this package to maintain backwards compatibility with.

## deep review of backwards-compat concerns

### 1. package rename

the repo was cloned from rhachet-brains-anthropic but renamed to rhachet-brains-bhrain.

**question**: did we add any shims to maintain anthropic compatibility?
**answer**: no. we deleted the anthropic-specific code (atoms/, repls/, hooks/) and replaced with bhrain code.
**verdict**: ✅ no backwards-compat cruft

### 2. api surface

**question**: did we add deprecated apis "to be safe"?
**files checked**: src/contract/sdk/index.ts

```ts
// what we export
export type { BrainEpisode, BrainOutput, BrainRepl } from 'rhachet';
export { genBrainRepl } from '../../domain.operations/brain.repl/genBrainRepl';
```

**answer**: no deprecated APIs. we export only `genBrainRepl` as specified in criteria.
**verdict**: ✅ clean api, no cruft

### 3. internal SDK wrappers (sdkAnthropic, sdkOpenAi, sdkQwen)

**question**: are these backwards-compat layers?

**files examined**:
- src/access/sdk/sdkOpenAi.ts
- src/access/sdks/anthropic/sdkAnthropic.ts
- src/access/sdks/qwen/sdkQwen.ts

**analysis**:

the blueprint stated: "BrainArch1Repl.atom accepts BrainAtom directly — no adapter needed"

this assumption was invalid. during execution, we discovered:
- rhachet's `BrainAtom.ask()` is high-level (prompt, schema, role)
- BrainArch1 needs low-level `generate(messages, tools, maxTokens)`

these interfaces are incompatible. the SDK wrappers provide the `generate()` interface that BrainArch1 requires.

**verdict**: ✅ NOT backwards-compat cruft — these are architectural necessities that emerged when the blueprint's assumption didn't hold. they were ejected from rhachet-roles-bhrain where they serve the same purpose.

### 4. direct SDK dependencies (@anthropic-ai/sdk, openai)

**question**: the criteria said "no direct dependencies on brain suppliers" — did we violate this?

**evidence**:
```json
// package.json dependencies
"@anthropic-ai/sdk": "0.39.0",
"openai": "4.103.0",
```

**analysis**:

the criteria said:
> "sothat this package has no direct dependencies on brain suppliers"

but we DO have direct SDK dependencies. this is because:
1. BrainArch1 needs `generate()` interface, not `ask()`
2. rhachet-brains-* packages only expose `ask()` via BrainAtom
3. to make BrainArch1 work, we need direct SDK access

**is this backwards compat?**: no. this is architectural necessity discovered during execution.

**open question for wisher**:
> the blueprint assumed we could use rhachet-brains-* peer deps. this didn't work because of the ask() vs generate() mismatch. should we:
> 1. accept direct SDK deps as necessary (current state)
> 2. refactor BrainArch1 to use ask() instead of generate()
> 3. request rhachet-brains-* to expose generate() interface

### 5. type compatibility shims

**question**: did we add type unions or any-casts for compatibility?

**files checked**: src/contract/sdk/index.test.ts

```ts
atom: mockAtom as any,  // line 26
slug: 'bhrain/arch1' as any,  // line 39

```

**answer**: these `as any` casts are for test mocks, not backwards compat shims. tests need to inject mock atoms that don't fully implement BrainArch1Atom.

**verdict**: ✅ not backwards-compat cruft

### 6. dependency version pins

**question**: did we pin old versions "to be safe"?
**answer**: versions are current/latest:
- rhachet: >=1.21.4 (peer dep)
- rhachet-brains-anthropic: >=0.3.0 (peer dep, for tests)
- rhachet-brains-openai: >=0.1.0 (peer dep, for tests)

**verdict**: ✅ no backwards-compat version pins

no backwards-compat cruft found. one deviation from blueprint documented as open question:
