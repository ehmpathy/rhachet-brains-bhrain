# self-review: has-questioned-assumptions

> surface hidden technical assumptions in the blueprint

---

## methodology

for each assumption, systematically ask:
1. what do we assume here without evidence?
2. what if the opposite were true?
3. is this architecture choice based on evidence or habit?
4. what exceptions or counterexamples exist?
5. could a simpler approach work?

---

## assumptions questioned

### 1. "imports will work after path rename"

**the assumption**: the rename from `@src/domain.operations/brain.replic.arch1` to `@src/domain.operations/arch1` via sedreplace will fix all imports.

**what do we assume without evidence?**
- all imports follow the `@src/` alias pattern
- no relative imports in BrainArch1 code
- no string literals contain the old path (e.g., in error messages, logs)

**what if opposite true?**
- some imports might use `../` relative paths
- some paths might be in comments or strings that we want preserved
- some paths might have variations like `@src/domain.operations/brain.replic.arch1/` (trailing slash)

**evidence gathered**:
```bash
# checked for relative imports in source repo
grep -r "from '\.\." in ejected directories: found none
grep -r "@src/" in ejected directories: all imports use alias
```

**is this evidence or habit?** evidence-based — grep confirmed pattern consistency

**exceptions?** none found

**simpler approach?** sedreplace with `--mode plan` first shows exact changes before apply

**verdict**: VERIFIED — import paths are consistent, sedreplace will work.

### 2. "68 files is the right scope"

**the assumption**: after removal of architect briefs (28) and brain role (4), the 68 files left are correct.

**what do we assume without evidence?**
- 28 briefs are all role-specific (not brain infrastructure)
- 4 brain role files belong in roles repo not brains repo
- the 68 files are pure brain infrastructure

**what if opposite true?**
- some briefs might document brain architecture (should be ejected)
- brain role might have brain-specific code (should be ejected)
- some of 68 might be role-specific (should NOT be ejected)

**evidence gathered**:
- briefs path: `.agent/repo=.this/role=architect/briefs/` — clearly role=architect specific
- brain role path: `src/domain.roles/brain/` — "roles" is the domain, belongs in roles repo
- the 68: domain.objects/BrainArch1, domain.operations/arch1, access/sdk(s), .test/

**is this evidence or habit?** evidence-based — directory structure makes ownership clear

**exceptions?** none — the path naming convention (`role=architect`, `domain.roles`) makes boundaries explicit

**simpler approach?** current approach is already minimal — delete role content, keep brain content

**verdict**: VERIFIED — 68 files represent pure brain infrastructure.

### 3. "dependencies can be added post-ejection"

**the assumption (original)**: blueprint said add deps (as-procedure, fast-glob, openai) in phase 4.

**what do we assume without evidence?**
- ejected code will compile without deps present
- typescript type checking can be deferred

**what if opposite true?**
- imports for absent deps will fail immediately after rsync
- `npm run build` in phase 5 would fail before we add deps
- safer to have deps present before importing code that uses them

**is this evidence or habit?** habit — "add deps at end" is common but not optimal here

**simpler approach?** add deps first (phase 2), then eject. ensures:
- no compilation errors during ejection
- imports validated immediately
- fails fast if deps incompatible

**action taken**: updated blueprint to add deps in phase 2 (before prune/eject).

**verified**: blueprint now shows deps in phase 2.

**verdict**: ADJUSTED — dependency addition moved to phase 2 for safety.

### 6. "hooks adapter is independent"

**the assumption**: genBrainHooksAdapterForClaudeCode doesn't depend on BrainArch1.

**what do we assume without evidence?**
- hooks code doesn't import from BrainArch1 directories
- no circular dependencies between hooks and arch1

**what if opposite true?**
- hooks might import BrainArch1 types
- hooks might use arch1 utilities

**evidence gathered**:
```
hooks location: src/domain.operations/hooks/
arch1 location: src/domain.operations/arch1/ (to be ejected INTO)
```
- hooks are in this repo already (anthropic template)
- arch1 is being ejected FROM roles repo
- no pre-extant dependency because arch1 doesn't yet exist here

**is this evidence or habit?** evidence — structural analysis shows independence

**exceptions?** after ejection, hooks COULD import from arch1 if we wanted — but that's a choice, not a requirement

**simpler approach?** retain hooks as-is, no changes needed

**verdict**: VERIFIED — hooks are independent, will coexist without conflict.
